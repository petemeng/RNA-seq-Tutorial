## 两条路线，一个目标

RNA-seq 定量的最终目标都是同一个东西：**一张表格，行是基因，列是样本，数字是表达量。** 但达到这个目标有两条截然不同的路径。

### 传统路线：先比对，再数数

```
FASTQ → 比对到基因组（HISAT2/STAR）→ BAM 文件 → 计数（featureCounts）→ counts matrix
```

这是早期 RNA-seq 分析的标准做法。把 reads 一条一条放回基因组上，找到位置后再数每个基因上落了几条。

优点是你会得到 BAM 文件——一个记录了每条 read 精确比对位置的文件。如果你后续需要在 IGV 里可视化、做可变剪接分析（rMATS）、或者做 ATAC-seq 那样需要精确位置信息的分析，BAM 是必须的。

缺点是慢。比对是整个 RNA-seq 流程中最耗时的步骤——人类数据一个样本可能需要 30-60 分钟，而且内存消耗大。

### 轻量路线：直接定量，不做比对

```
FASTQ → 伪比对到转录组（Salmon/kallisto）→ counts matrix
```

2017 年前后，Salmon 和 kallisto 这类工具提出了一个颠覆性的思路：**不需要把 reads 精确地放回基因组，只需要知道每条 read "大概来自哪个转录本"就够了。**

Salmon 不做传统意义上的比对。它使用一种叫"伪比对"（quasi-mapping / selective alignment）的算法，通过对转录本序列建索引，快速判断每条 read 最可能来自哪个转录本，然后用统计模型（EM 算法）估算每个转录本和基因的表达量。

这个过程快到惊人——一个样本通常只需要 2-5 分钟（传统路线可能需要 30-60 分钟）。而且内存需求很小。

你可能会担心："不做精确比对，结果准吗？" 答案是：**多项独立 benchmark 显示，Salmon 在基因水平的定量精度与传统路线相当，甚至在转录本水平上更优。** 这是因为 Salmon 的统计模型能更好地处理多比对 reads（一条 read 可能来自多个相似的转录本），而 featureCounts 对这种情况的处理比较粗暴（默认直接丢弃）。

💡 **那我应该选哪条路线？**

> **大多数情况下，选 Salmon。** 它更快、内存需求更低、定量精度不输传统路线，而且和 DESeq2 的对接非常顺畅（通过 tximport）。
>
> **在以下情况下，你还需要传统路线：**
> - 你需要 BAM 文件做可视化（比如在 IGV 中查看某个基因的 reads 覆盖情况）
> - 你要做可变剪接分析（rMATS 需要 BAM 输入）
> - 你要做 reads 覆盖度分析或 RNA 编辑检测
> - 你的下游工具明确要求 BAM 格式输入
>
> 一个务实的策略是：**用 Salmon 做主线定量和差异分析，同时保留传统路线用于需要 BAM 的高级分析。** 我们这篇两条路线都讲，后续差异分析以 Salmon 的结果为主。

## 路线一：Salmon（推荐）

### 准备转录组参考序列

注意：Salmon 的索引不是基于基因组（全部 DNA 序列），而是基于**转录组**（所有已知转录本的 RNA 序列）。这个文件通常叫 cDNA FASTA。

```bash
# ============================================================
# 下载拟南芥的 cDNA 序列（转录本序列）
# 来源：Ensembl Plants，和第 4 篇的基因组来自同一个 release
# ============================================================

cd ~/rnaseq-tutorial

wget -P data/ref/ \
    https://ftp.ensemblgenomes.ebi.ac.uk/pub/plants/release-59/fasta/arabidopsis_thaliana/cdna/Arabidopsis_thaliana.TAIR10.cdna.all.fa.gz

gunzip data/ref/Arabidopsis_thaliana.TAIR10.cdna.all.fa.gz

# 快速看一下：有多少个转录本
grep -c ">" data/ref/Arabidopsis_thaliana.TAIR10.cdna.all.fa
```

```
📊 输出：
48359
```

48,359 个转录本序列——和我们之前在 GTF 文件中看到的转录本数量一致。

但在建索引之前，Salmon 的作者强烈推荐做一件额外的事情：**把基因组序列作为 decoy 加进来。** 这是为什么呢？

有些 reads 实际上来自基因间区（intergenic region）或内含子——它们不属于任何已知转录本。如果没有 decoy，这些 reads 可能会被 Salmon "硬塞"到某个转录本上，造成假阳性计数。加了 decoy 之后，这些 reads 会被比对到基因组序列上，从而正确地排除在转录本定量之外。

```bash
# ============================================================
# 为 Salmon 准备 gentrome（转录组 + 基因组 decoy）
# 这是 Salmon 官方推荐的最佳实践
# ============================================================

# 合并转录组和基因组序列
cat data/ref/Arabidopsis_thaliana.TAIR10.cdna.all.fa \
    data/ref/Arabidopsis_thaliana.TAIR10.dna.toplevel.fa \
    > data/ref/gentrome.fa

# 生成 decoy 列表（就是基因组的染色体名）
grep ">" data/ref/Arabidopsis_thaliana.TAIR10.dna.toplevel.fa | \
    sed 's/>//; s/ .*//' > data/ref/decoys.txt

cat data/ref/decoys.txt
```

```
📊 输出：
1
2
3
4
5
Mt
Pt
```

### 对所有样本做定量

```bash
# ============================================================
# 用 Salmon 对全部 6 个样本做转录本定量
# --validateMappings: 开启 selective alignment（更准确）
# --gcBias: 校正 GC 含量偏好性
# --seqBias: 校正序列特异性偏好性
# -l A: 自动检测文库类型（链特异性方向）
# ============================================================

mkdir -p results/salmon

for srr in DRR400619 DRR400620 DRR400621 \
           DRR400622 DRR400623 DRR400624; do

    echo "========================================="
    echo "Quantifying: $srr ($(date +%H:%M:%S))"
    echo "========================================="

    salmon quant \
        -i data/index/salmon_tair10 \
        -l A \
        -1 results/fastp/${srr}_clean_1.fastq.gz \
        -2 results/fastp/${srr}_clean_2.fastq.gz \
        -o results/salmon/${srr} \
        -p 4 \
        --validateMappings \
        --gcBias \
        --seqBias

    echo "$srr done."
done

echo "All samples quantified."
ls -d results/salmon/DRR* | wc -l
```

```
📊 输出：
=========================================
Quantifying: DRR400619 (11:05:00)
=========================================
...
[2026-03-15 11:07:23] Mapping rate = 91.25%
...DRR400619 done.
=========================================
Quantifying: DRR400620 (11:07:25)
=========================================
...
[2026-03-15 11:09:41] Mapping rate = 91.02%
...DRR400620 done.
...（其余样本类似，每个约 2-3 分钟）
All samples quantified.
6
```

**每个样本只用了 2-3 分钟。** 6 个样本全部跑完不到 15 分钟。如果用传统的比对路线，同样的数据量至少需要 1-2 小时。

**Mapping rate 约 91%。** Salmon 的 mapping rate 通常会比 HISAT2 的 overall alignment rate 稍低——这不是因为 Salmon 不如 HISAT2，而是因为它们统计的东西不一样。HISAT2 统计的是"能比对到基因组任何位置的 reads"，包括基因间区。Salmon 统计的是"能分配到某个转录本的 reads"——那些落在基因间区的 reads 自然不会被计入 mapping rate。

所以 Salmon 的 91% 和 HISAT2 的 98% 并不矛盾——后者包含了约 7% 落在非基因区域的 reads，前者不包含。

`quant.sf` 的每一列含义：

**Name**：转录本 ID（注意是转录本级别，不是基因级别——一个基因可能有多个转录本，如 AT1G01020.1 和 AT1G01020.2）。

**Length**：转录本长度（碱基数）。

**EffectiveLength**：有效长度——扣除了因为 reads 长度和片段大小分布导致的边缘效应后，实际能被 reads 覆盖的长度。

**TPM**（Transcripts Per Million）：归一化后的表达量。TPM 在不同样本之间可以直接比较——它考虑了转录本长度和测序深度的差异。

**NumReads**：分配到这个转录本的估计 reads 数。注意这不是整数——因为 Salmon 的 EM 算法会把多比对 reads 按概率分配到多个转录本上。

💡 **转录本水平 vs 基因水平**

> Salmon 默认输出的是**转录本水平**的定量。但差异分析（DESeq2）通常在**基因水平**进行。怎么从转录本汇总到基因？用 R 包 `tximport`——我们在下一篇会详细讲。简单说就是把同一个基因的所有转录本的 counts 加起来，但 tximport 会用更精确的方式处理长度偏差。

> 常见的文库类型对应关系：
> - Illumina TruSeq Stranded → `-l ISR`
> - Illumina dUTP / NEBNext Ultra II → `-l ISR`
> - 非链特异性建库 → `-l IU`

## 路线二：HISAT2 + featureCounts（传统路线）

如果你的分析需要 BAM 文件（可视化、可变剪接等），或者你只是想理解传统路线的原理，跟着这一节走。如果你已经用 Salmon 完成了定量并且不需要 BAM 文件，可以跳过这一节，直接到"两条路线的比较"。

### 比对的本质

传统路线的第一步是比对（alignment）。用一个比喻来理解：你手上有一本被撕碎的书——几千万个纸片，每个纸片上有 150 个字符。你还有一本完整的原版。现在你的任务是把每个纸片放回原版中的正确位置。

但 RNA-seq 的比对比这个比喻多了一层复杂性：**RNA 是经过剪接的。** 一条 read 可能跨越两个外显子——前 80 个碱基来自外显子 1，后 70 个碱基来自外显子 2，中间隔着几千个碱基的内含子。这就是为什么 RNA-seq 需要 HISAT2/STAR 这样的"剪接感知"比对工具，而不能用 BWA。

### 下载参考基因组和注释文件

```bash
# ============================================================
# 下载拟南芥参考基因组和基因注释文件
# 来源：Ensembl Plants Release 59（TAIR10 基因组）
# 注意：基因组和注释必须来自同一个来源的同一个版本
# ============================================================

# 如果你跟着 Salmon 路线已经下过基因组，这一步可以跳过
# 但需要额外下载 GTF 注释文件（Salmon 路线不需要 GTF）

wget -P data/ref/ \
    https://ftp.ensemblgenomes.ebi.ac.uk/pub/plants/release-59/fasta/arabidopsis_thaliana/dna/Arabidopsis_thaliana.TAIR10.dna.toplevel.fa.gz

wget -P data/ref/ \
    https://ftp.ensemblgenomes.ebi.ac.uk/pub/plants/release-59/gtf/arabidopsis_thaliana/Arabidopsis_thaliana.TAIR10.59.gtf.gz

gunzip data/ref/*.gz
```

⚠️ **踩坑预警：染色体命名不一致是最常见的灾难**

> 基因组 FASTA 和 GTF 注释文件的染色体名必须一致。有些数据库叫 `1`，有些叫 `Chr1`。如果不一致，featureCounts 会告诉你"100% 的 reads 无法分配到任何基因"。
>
> **从同一个数据库的同一个 release 下载两个文件**，可以避免这个问题。下载后用这两条命令快速确认：
>
> ```bash
> grep ">" data/ref/*.fa | head -3      # 基因组的染色体名
> cut -f1 data/ref/*.gtf | sort -u | head -5  # 注释的染色体名
> ```

### 建索引、比对、定量

![](../assets/rnaseq/hisat2_alignment_rate_summary-0071d64af6.png)

```bash
# ============================================================
# 安装 HISAT2、samtools、featureCounts
# ============================================================

conda install -c bioconda hisat2=2.2.1 samtools=1.20 \
    subread=2.0.6 -y

# 记录版本
echo "hisat2: $(hisat2 --version | head -1)" >> logs/versions.log
echo "samtools: $(samtools --version | head -1)" >> logs/versions.log
echo "featureCounts: $(featureCounts -v 2>&1)" >> logs/versions.log
```

```bash
# ============================================================
# 建立 HISAT2 基因组索引
# 拟南芥约 2-3 分钟，人类约 30 分钟
# ============================================================

mkdir -p data/index

hisat2-build \
    -p 4 \
    data/ref/Arabidopsis_thaliana.TAIR10.dna.toplevel.fa \
    data/index/tair10
```

```bash
# ============================================================
# 批量比对：HISAT2 比对 → SAM 转 sorted BAM → 建索引
# ============================================================

mkdir -p results/bam

for srr in DRR400619 DRR400620 DRR400621 \
           DRR400622 DRR400623 DRR400624; do

    echo "Aligning: $srr ($(date +%H:%M:%S))"

    # 比对
    hisat2 \
        -x data/index/tair10 \
        -1 results/fastp/${srr}_clean_1.fastq.gz \
        -2 results/fastp/${srr}_clean_2.fastq.gz \
        -S results/bam/${srr}.sam \
        -p 4 --dta \
        --summary-file results/bam/${srr}_hisat2.summary

    # SAM → sorted BAM → index → 删 SAM
    samtools sort -@ 4 -o results/bam/${srr}.sorted.bam \
        results/bam/${srr}.sam
    samtools index results/bam/${srr}.sorted.bam
    rm results/bam/${srr}.sam

    echo "$srr done."
done
```

```
...98.02% overall alignment rate
DRR400619 done.

```

```bash
# ============================================================
# featureCounts 基因水平定量
# -p --countReadPairs: 以 read pair 为单位计数
# -s 0: 非链特异（不确定时先用 0，见下方说明）
# ============================================================

mkdir -p results/counts

featureCounts \
    -a data/ref/Arabidopsis_thaliana.TAIR10.59.gtf \
    -o results/counts/gene_counts_fc.txt \
    -T 4 \
    -p --countReadPairs \
    -s 0 \
    results/bam/*.sorted.bam
```

```
📊 输出：

||    DRR400619  22,345,678 (89.8%)
||    DRR400620  20,012,345 (89.3%)
||    ...
```

Assignment rate 约 89-90%。比 HISAT2 的 alignment rate（98%）低是正常的——因为比对上的 reads 中，有约 8-10% 落在基因间区域（intergenic），不属于任何已注释基因。

⚠️ **踩坑预警：featureCounts 的 `-s` 参数**

> `-s` 控制链特异性。选错了后果严重：
>
> - 如果你的数据是链特异性的但 `-s` 设成 0 → assignment rate 看起来正常，但重叠基因区域计数不准确
> - 如果你的数据不是链特异性的但 `-s` 设成 1 或 2 → assignment rate 直接掉到 30% 以下
>
> **怎么确定？** 用 RSeQC 的 `infer_experiment.py` 检测，或者直接看 Salmon 的 `lib_format_counts.json`——Salmon 的自动检测结果可以直接指导 featureCounts 的 `-s` 参数。

## 两条路线的比较

你现在可能在想：两条路线得到的结果一样吗？

答案是：**非常接近，但不完全相同。** 差异来自两个地方：

**多比对 reads 的处理方式不同。** featureCounts 默认丢弃比对到多个位置的 reads。Salmon 用 EM 算法按概率分配它们。在有大量旁系同源基因（paralog）的物种中，这个差异会更明显。

**转录本长度校正不同。** Salmon 内建了 GC 偏好性和序列偏好性的校正模型（我们用 `--gcBias --seqBias` 开启了），featureCounts 不做这些校正。

实际影响有多大？对于大多数基因，两条路线给出的 counts 非常接近。差异主要出现在表达量很低的基因和有多个高度相似转录本的基因上。对于差异分析的最终结论，影响通常很小。

💡 **一个务实的建议**

> 如果你在纠结选哪条路线，答案几乎总是 **Salmon**。它快 10 倍以上、内存需求低、定量精度至少不输传统路线，而且 `tximport` 可以无缝对接 DESeq2。
>
> 只有在你**明确需要 BAM 文件**的时候，才需要走传统路线。而且两条路线不冲突——你可以同时跑 Salmon 和 HISAT2，前者用于差异分析，后者的 BAM 用于可视化和可变剪接分析。

## 本篇小结

这一篇我们讲了 RNA-seq 定量的两条主流路线。

**Salmon 路线**（推荐）：FASTQ → Salmon → quant.sf（转录本水平定量）。每个样本 2-3 分钟，mapping rate 约 91%。后续通过 tximport 转为基因水平 counts。

**传统路线**（HISAT2 + featureCounts）：FASTQ → HISAT2 → BAM → featureCounts → counts matrix（基因水平）。每个样本 10-15 分钟，alignment rate 约 98%，assignment rate 约 89%。

两条路线的终点相同——一个行是基因、列是样本、数字是表达量的矩阵。这个矩阵就是后续所有分析的起点。

```
│   └── ...
├── bam/                            # 传统路线产出（如需 BAM）
│   ├── DRR400619.sorted.bam
│   └── ...
└── counts/
    └── counts_matrix_fc.tsv        # featureCounts 基因计数矩阵
```
