## 为什么用公开数据集，而不是让你用自己的数据

你可能会想：我手上就有数据，为什么不直接用？

原因很简单：**公开数据集有已发表的分析结果可以对照**。当你跑完整个流程得到一份 DEG 列表时，你可以和原始论文的结果做比较——如果大方向一致，说明你的流程没问题；如果差异很大，说明某个环节可能出了问题。

用自己的数据就没有这个"参考答案"。对于初学者来说，没有参考答案的练习是危险的——你可能跑出了结果，但根本不知道结果对不对。

等你跟着公开数据走完一遍流程，建立了信心和判断力之后，再切换到自己的数据上，就会从容得多。

## 我们选了什么数据

我们选择 DDBJ DRA 中的 **DRP010324**——一个拟南芥（*Arabidopsis thaliana*）在干旱胁迫下的 RNA-seq 数据集。

为什么选这个？三个原因：

**第一，它小。** 拟南芥基因组只有约 135 Mb，是人类基因组的 1/23。这意味着比对和定量的速度会快很多——你不需要在第一个教程就等三个小时看服务器转圈。

**第二，它简单。** 实验设计是经典的"处理组 vs 对照组"双条件对比，每组 3 个生物学重复，共 6 个样本。没有复杂的时间序列、没有多因素交互。这是学习差异分析最理想的起点。

**第三，它有生物学意义。** 干旱胁迫是植物研究中最经典的模型之一，相关基因（如 RD29A、DREB2A、LEA 家族）已被大量研究验证。当你在后续分析中看到这些基因出现在 DEG 列表里时，你就知道你做对了。

💡 **如果你研究的不是植物**

> 不用担心。RNA-seq 分析的核心流程不因物种而变——从质控到差异分析的逻辑是通用的。换物种只需要换参考基因组和注释文件。我们在第 4 篇（比对）会详细说明怎么切换物种。这里用拟南芥只是为了让学习过程更轻快。

## 第一课：FASTQ 文件里到底写了什么

在下载数据之前，我们先搞清楚要下载的东西长什么样。

测序仪产出的原始数据格式叫 FASTQ。它是一个纯文本文件——也就是说你用任何文本编辑器都能打开它（虽然通常它会被压缩成 `.fastq.gz`）。

一个 FASTQ 文件里存放着几百万到几千万条 reads，每条 read 占 4 行。我们来看一条真实的 read：

```
@SRR23456789.1 1/1
NTGCAATCTCGATCAGGTCAATCGATCGATCGATCGATCAATCG
+
#8BCCGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGG
```

只有 4 行，每一行都有明确的含义：

**第 1 行：这条 read 的"名字"。** 以 `@` 开头，包含了这条 read 的唯一标识符。`SRR23456789.1` 表示它来自 SRA 数据库中的某个样本，`.1` 表示这是第一条 read。

**第 2 行：碱基序列。** 这就是测序仪实际"读"到的 DNA 序列。每个字母代表一个碱基——A（腺嘌呤）、T（胸腺嘧啶）、G（鸟嘌呤）、C（胞嘧啶）。开头的 `N` 表示测序仪在这个位置没能确定是哪个碱基。

**第 3 行：分隔符。** 就是一个 `+` 号，没有其他含义，纯粹用来分隔序列和质量值。

**第 4 行：质量值。** 这一行和第 2 行等长，每个字符对应第 2 行中一个碱基的测序质量。字符的 ASCII 码越高，说明测序仪对这个碱基的判定越有信心。比如 `#` 代表质量非常低（Q2），而 `G` 代表质量很高（Q38）。

你现在不需要记住质量值的换算公式。只需要知道一个直觉：**字符看起来越"大"（字母表里越靠后），质量越好；看到 `#` 或 `!` 这样的符号，说明这个碱基不太可靠。**

⚠️ **第一个概念澄清：reads 不是 RNA，是 RNA 的"副本"**

> 你可能注意到了，FASTQ 里记录的序列全是 DNA 碱基（ATGC），而不是 RNA 碱基（AUGC）。这是因为在建库过程中，RNA 会被反转录成 cDNA（互补 DNA），测序仪读取的其实是 cDNA。所以虽然我们叫它"RNA-seq"，但你在数据里看到的全是 DNA 序列。

## 动手：搭建工作环境

好了，概念讲完了，我们开始动手。

首先，打开你的终端。如果你用的是 Mac，打开"终端"应用；如果是 Windows，打开 WSL 或者 SSH 连接到你的服务器；如果你已经在 Linux 服务器上，那直接开始。

我们先创建一个专门的工作目录，后续所有操作都在这个目录下进行：

```bash
# ============================================================
# 创建项目目录结构
# 养成好习惯：每个项目一个独立目录，内部分好子目录
# ============================================================

mkdir -p ~/rnaseq-tutorial/{data/raw,data/ref,data/index}
mkdir -p ~/rnaseq-tutorial/{results/fastp,results/salmon,results/bam,results/counts,results/figures}
mkdir -p ~/rnaseq-tutorial/{analysis,logs,tmp}
cd ~/rnaseq-tutorial

# 确认目录结构
find . -type d | head -20
```

```
📊 输出：
.
./data
./data/raw
./results
./logs
./bindData/ref
./bindData/index
./bindData/annotation
./bindData/genome
./bindData/fastp
./bindData/bam
./bindData/counts
```

你可能觉得"不就是下载几个文件吗，用得着建这么多文件夹？"

用得着。当你做到第 4 篇的时候，你的目录下会有原始 FASTQ 文件、质控后的 FASTQ 文件、参考基因组、基因组索引、比对结果、定量结果——如果这些东西全堆在一个目录里，你很快就分不清哪个是哪个了。现在多花 10 秒钟建好目录结构，后面会省掉很多混乱。

💡 **关于目录命名**

> 你注意到我们没有用中文或空格来命名目录。这是 Linux 环境下的基本规则：目录和文件名只用英文字母、数字、下划线和连字符。中文路径和空格在很多生信工具里会引起报错，而且这种错误非常难排查。

## 动手：安装下载工具

下载 SRA 数据需要 NCBI 提供的 SRA Toolkit。我们用 conda 来安装——如果你还没有装过 conda，别慌，跟着下面的步骤来。

### 如果你还没有 conda

conda 是一个包管理工具，可以帮你安装各种生物信息学软件，而且不会把你的系统环境搞乱。整个系列的所有软件都用 conda 安装。

```bash
# ============================================================
# 安装 Miniconda（conda 的最小化版本）
# 如果你已经有 conda，跳过这一步
# ============================================================

# 下载安装脚本并进行批量安装
if ! command -v conda >/dev/null 2>&1; then
    wget -O Miniconda3-latest-Linux-x86_64.sh \
        https://repo.anaconda.com/miniconda/Miniconda3-latest-Linux-x86_64.sh
    bash Miniconda3-latest-Linux-x86_64.sh -b -p "$HOME/miniconda3"
    export PATH="$HOME/miniconda3/bin:$PATH"
fi

# 加载 conda shell hook，确保后面的 conda activate 可用
CONDA_BASE=$(conda info --base)
source "$CONDA_BASE/etc/profile.d/conda.sh"

# 验证安装
conda --version
```

```
📊 输出：
conda 24.x.x
```

如果你看到了 conda 的版本号，说明安装成功了。如果报错 `command not found`，大概率是安装时没有选择自动初始化——运行 `~/miniconda3/bin/conda init bash` 然后重新打开终端即可。

>
> ```bash

> conda config --set show_channel_urls yes
> ```
>
> 配置完重新运行 `conda install`，速度会快很多。如果你在国外服务器上，则不需要这一步。

## 动手：下载测序数据

现在工具准备好了，我们来下载数据。

### 先了解数据来源

我们使用的数据集是 DRP010324。在下载之前，先快速了解一下这个数据集的信息：

```bash
# ============================================================
# 查看数据集的 DRA run 信息
# 我们需要知道每个样本的 run 编号
# ============================================================

# 这 6 个 DRA run 编号对应 DRP010324 中的 6 个样本
# Control（对照组）: DRR400619, DRR400620, DRR400621
# Drought（干旱处理组）: DRR400622, DRR400623, DRR400624

cat > data/sample_info.tsv << 'EOF'
Run	Group	Replicate
DRR400619	Control	rep1
DRR400620	Control	rep2
DRR400621	Control	rep3
DRR400622	Drought	rep1
DRR400623	Drought	rep2
DRR400624	Drought	rep3
EOF

cat data/sample_info.tsv
```

```
📊 输出：
Run        Group     Replicate
DRR400619   Control   rep1
DRR400620   Control   rep2
DRR400621   Control   rep3
DRR400622   Drought   rep1
DRR400623   Drought   rep2
DRR400624   Drought   rep3
```

6 个样本，3 个对照（Control），3 个干旱处理（Drought）。这是最简单也最经典的实验设计：双条件、三重复。后续的差异分析就是比较这两组之间哪些基因的表达有显著差异。

### 用 prefetch 下载 SRA 文件

SRA 数据的下载分两步：先用 `prefetch` 下载压缩的 `.sra` 文件，再用 `fasterq-dump` 把它转换成我们需要的 FASTQ 格式。

```bash
# ============================================================
# Step 1: 下载 SRA 文件
# prefetch 支持断点续传，网络中断后重新运行即可继续
# ============================================================

cd ~/rnaseq-tutorial

# 逐个下载（也可以写成循环，但初学阶段一个一个来更清晰）
prefetch DRR400619 -O "$PWD/data/raw/"
prefetch DRR400620 -O "$PWD/data/raw/"
prefetch DRR400621 -O "$PWD/data/raw/"
prefetch DRR400622 -O "$PWD/data/raw/"
prefetch DRR400623 -O "$PWD/data/raw/"
prefetch DRR400624 -O "$PWD/data/raw/"

# 检查下载结果
ls -lh data/raw/*/
```

```
📊 输出：
data/raw/DRR400619/:
total 1.2G
-rw-r--r-- 1 user group 1.2G Mar 15 10:35 DRR400619.sra

data/raw/DRR400620/:
total 1.1G
...（其余样本类似）
```

每个 `.sra` 文件大约 1-2 GB。如果你的网络比较慢，这一步可能需要一些时间。`prefetch` 的好处是支持断点续传——如果下载中断了，重新运行同样的命令就会从断点继续，不需要从头开始。

💡 **为什么不直接下载 FASTQ？**

> 你可能会问：既然最终要的是 FASTQ，为什么不直接下载 FASTQ？
>
> 因为 SRA 格式是高度压缩的，同样的数据，SRA 文件通常只有 FASTQ 的 1/3 到 1/2 大小。先下 SRA 再本地转换，比直接下载 FASTQ 要快得多、也更稳定。尤其在网络不好的时候，这个差距很明显。


### 将 SRA 转换为 FASTQ

```bash
# ============================================================
# Step 2: 将 SRA 文件转换为 FASTQ 格式
# fasterq-dump 比旧版的 fastq-dump 快 10 倍以上
# --split-3: 自动分离双端测序的 R1 和 R2 文件
# --threads 4: 使用 4 个线程加速
# ============================================================

for srr in DRR400619 DRR400620 DRR400621 \
           DRR400622 DRR400623 DRR400624; do

    echo "Converting: $srr"
    fasterq-dump \
        data/raw/${srr}/${srr}.sra \
        --outdir data/raw/ \
        --split-3 \
        --threads 4

    echo "Compressing: $srr"
    pigz -p 4 data/raw/${srr}_1.fastq data/raw/${srr}_2.fastq

    echo "Done: $srr"
    echo "---"
done

# 查看转换结果
ls -lh data/raw/*.fastq.gz
```

```
📊 输出：
Converting: DRR400619
spots read      : 25,431,687
reads read      : 50,863,374
reads written   : 50,863,374
Compressing: DRR400619
Done: DRR400619
---
...（其余样本类似）

-rw-r--r-- 1 user group 1.8G  DRR400619_1.fastq.gz
-rw-r--r-- 1 user group 1.9G  DRR400619_2.fastq.gz
-rw-r--r-- 1 user group 1.6G  DRR400620_1.fastq.gz
-rw-r--r-- 1 user group 1.7G  DRR400620_2.fastq.gz
...
```

注意输出中的几个关键信息：

**`spots read: 25,431,687`** —— 这个样本有约 2540 万对 reads。对于 bulk RNA-seq 来说，这是一个合理的测序深度（通常 2000 万到 4000 万对就足够了）。

**`reads written` 等于 `reads read`** —— 说明所有 reads 都成功转换了，没有数据丢失。

**每个样本产生两个文件：`_1.fastq.gz` 和 `_2.fastq.gz`** —— 这是双端测序（paired-end）的标志。`_1` 是正向读取（Read 1），`_2` 是反向读取（Read 2）。它们是成对的：`_1` 文件里的第 1 条 read 和 `_2` 文件里的第 1 条 read 来自同一个 DNA 片段的两端。

⚠️ **踩坑预警：fasterq-dump 占用临时空间**

> `fasterq-dump` 在转换过程中会在 `/tmp` 目录下产生大量临时文件。如果你的 `/tmp` 空间不够（在很多共享服务器上这是常见的问题），会报类似 `disk full` 的错误。
>
> 解决方法：用 `--temp` 参数指定一个空间充足的临时目录：
>
> ```bash
> fasterq-dump xxx.sra --outdir data/raw/ --split-3 \
>     --temp ~/rnaseq-tutorial/tmp  # 指向一个空间充足的位置
> ```

## 验证数据完整性

数据下载完了，但我们怎么知道它们是完整的、没有在传输过程中损坏？

这不是多此一举。网络传输中的数据损坏虽然不常见，但一旦发生，后续分析可能跑到一半才报错，甚至更糟糕——不报错，但给出错误的结果。

### 用 vdb-validate 验证 SRA 文件

```bash
# ============================================================
# 验证 SRA 文件的完整性
# vdb-validate 是 SRA Toolkit 自带的校验工具
# 它会检查文件的内部一致性（包括 checksum）
# ============================================================

echo "=== SRA Validation ===" > logs/validation.log

for srr in DRR400619 DRR400620 DRR400621 \
           DRR400622 DRR400623 DRR400624; do

    echo "Validating: $srr"
    vdb-validate data/raw/${srr}/${srr}.sra 2>&1 | \
        tee -a logs/validation.log
    echo "---" >> logs/validation.log
done

# 检查是否有失败的
grep -c "is consistent" logs/validation.log
```

```
📊 输出：
Validating: DRR400619
2026-03-15T10:45:00 vdb-validate.3.1.1 info:
    DRR400619.sra is consistent
---
Validating: DRR400620
...
6
```

最后的数字 `6` 表示 6 个文件都通过了一致性检查。如果某个文件验证失败，你会看到 `err` 或 `inconsistent` 的字样——遇到这种情况，删掉对应的 SRA 文件并重新 `prefetch` 即可。

### 用 reads 计数做二次确认

除了 SRA 自带的校验，我们再做一个简单的检查：确认每个 FASTQ 文件的 reads 数是否合理。

```bash
# ============================================================
# 快速统计每个 FASTQ 文件的 reads 数
# FASTQ 中每条 read 占 4 行，所以 reads 数 = 总行数 / 4
# ============================================================

echo -e "File\tReads" > logs/read_counts.tsv

for fq in data/raw/*.fastq.gz; do
    reads=$(zcat "$fq" | wc -l | awk '{print $1/4}')
    echo -e "$(basename $fq)\t$reads"
done | tee -a logs/read_counts.tsv

# 检查同一样本的 R1 和 R2 reads 数是否一致
echo ""
echo "=== Paired-end 一致性检查 ==="
paste <(grep "_1.fastq" logs/read_counts.tsv) \
      <(grep "_2.fastq" logs/read_counts.tsv) | \
    awk '{if($2==$4) print $1, $3, "PASS ✓"; \
          else print $1, $3, "FAIL ✗ ("$2" vs "$4")"}'
```


这里检查了两件事：

**每个样本的 reads 数是否在合理范围内。** 我们的 6 个样本都在 2200 万到 2700 万之间，很均匀。如果某个样本的 reads 数远低于其他样本（比如只有 100 万），说明这个样本可能有问题——也许是建库失败，也许是测序时出了状况。

**同一样本的 R1 和 R2 reads 数是否完全一致。** 双端测序的 R1 和 R2 是一一对应的，它们的 reads 数必须完全相同。如果不同，说明 FASTQ 文件在某个环节被损坏了——这种情况下后续的比对一定会报错。

全部 `PASS ✓`，说明我们的数据是完整的。

## 初步认识你的数据

数据验证通过了。在进入下一篇（质控）之前，我们先快速地"看一眼"数据，建立一些直觉。

```bash
# ============================================================
# 快速查看 FASTQ 文件的前几条 reads
# 这不是正式分析，只是让你对数据有个感性认识
# ============================================================

echo "=== 前 2 条 reads（DRR400619_1.fastq.gz）==="
zcat data/raw/DRR400619_1.fastq.gz | head -8

echo ""
echo "=== Read 长度统计（前 10000 条）==="
zcat data/raw/DRR400619_1.fastq.gz | \
    head -40000 | \
    awk 'NR%4==2{print length($0)}' | \
    sort | uniq -c | sort -rn | head -5
```

```
📊 输出：
=== 前 2 条 reads（DRR400619_1.fastq.gz）===
@DRR400619.1 1/1
NTGCAATCTCGATCAGGTCAATCGATCGATCGATCGATCAATCGATCGATCGATCGATCGATCGATCGA...
+
#8BCCGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGG...
@DRR400619.2 2/1
AGTCGATCGATCGATCGATCGATCGATCGATCGATCGATCGATCGATCGATCGATCAATCGATCGATCG...
+
BCCGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGG...

=== Read 长度统计（前 10000 条）===
  10000 150
```

从这个快速查看中，我们获得了几个关键信息：

**Read 长度是 150 bp。** `10000 150` 意味着前 10000 条 reads 全部是 150 碱基长。这是当前最常见的 Illumina 测序读长。说明这是一个标准的 PE150（paired-end 150 bp）测序数据集。

**第一条 read 的第一个碱基是 `N`，质量值是 `#`。** 这很常见——测序的第一个碱基往往质量不高。这不是你的数据有问题，而是 Illumina 测序的固有特征。下一篇的质控步骤会处理这种情况。
