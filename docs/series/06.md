## 先搞清楚一件事：ORA 和 GSEA 不是一回事

很多教程把"GO 富集分析"和"GSEA"混为一谈，或者只讲一个不提另一个。但它们的逻辑完全不同，适用场景也不同。

### ORA（Over-Representation Analysis）：数盒子里有多少球

ORA 的逻辑很简单。你有一份 DEG 列表（比如 3,400 个基因），你问：**这些基因里，属于某个功能类别（比如"光合作用"）的基因是不是比随机预期的多？**

类比：一个袋子里有 100 个球，其中 20 个红色球。你闭眼摸出 10 个球，结果有 7 个红色的。正常情况下你只"应该"摸到 2 个红色球——7 个远超预期，说明红色球在你摸出的这批里"过度代表"（over-represented）了。

ORA 就是这个检验。"袋子"是全部基因，"摸出的球"是你的 DEG 列表，"红色球"是属于某个 GO term 或 KEGG pathway 的基因。

**ORA 的问题：它需要一个硬阈值来定义 DEG。**

你必须先决定"padj < 0.05 且 |log2FC| > 1"才算差异基因——但这个阈值是人为的。一个基因 padj = 0.049 被选入 DEG，另一个 padj = 0.051 被排除，但它们的差异表达证据几乎一模一样。ORA 对这种"门槛效应"很敏感。

### GSEA（Gene Set Enrichment Analysis）：不设门槛，看趋势

GSEA 的思路完全不同。它不需要你定义 DEG——它用的是**全部基因**的排名列表。

具体来说：你把所有基因按照差异表达的统计量（比如 DESeq2 的 `stat` 列或 `log2FoldChange`）从大到小排序，得到一个排名列表。然后 GSEA 检查：**属于某个功能类别的基因，是不是倾向于聚集在排名的顶部或底部？**

如果"光合作用"相关的基因大部分排在列表的最前面（最上调的那些），就说明整个光合作用通路是协同上调的，即使其中有些基因单独看并不"显著"。

**GSEA 的优势是不依赖任何阈值。** 它看的是整个通路的集体行为，而不是单个基因的 p 值。

💡 **那我应该用哪个？**

> **两个都做。** ORA 简单直接，结果容易解释，适合在论文中作为第一层展示。GSEA 更严谨，能发现 ORA 遗漏的通路（尤其是效应温和但方向一致的通路），适合作为补充验证。
>
> 如果两者结论一致，你的结果就更有说服力。如果不一致，值得深入探究原因。

```r
load_annotation_map <- function(cache_file, mart, attributes) {
    if (file.exists(cache_file)) {
        cat("读取本地 biomaRt 注释缓存:", cache_file, "\n")
        annotation_map <- read_tsv(cache_file, show_col_types = FALSE)
    } else {
        cat("下载全量拟南芥 GO 注释并写入缓存...\n")
        annotation_map <- getBM(attributes = attributes, mart = mart)
        write_tsv(annotation_map, cache_file)
    }

    annotation_map
}

build_go_tables <- function(annotation_map) {
    keep <- !is.na(annotation_map$go_id) &
        annotation_map$go_id != "" &
        annotation_map$namespace_1003 == "biological_process"
    go_bp_map <- unique(annotation_map[keep, c("go_id", "ensembl_gene_id")])

    term2gene <- data.frame(
        ID = go_bp_map$go_id,
        Gene = go_bp_map$ensembl_gene_id
    )

    term_ids <- sort(unique(term2gene$ID))
    term2name <- data.frame(
        ID = term_ids,
        Description = vapply(
            term_ids,
            function(id) {
                term <- Term(id)
                if (is.null(term) || is.na(term)) id else term
            },
            character(1)
        )
    )

    list(term2gene = term2gene, term2name = term2name)
}
```

## Part A：GO 过表达分析（ORA）

### 什么是 GO

GO（Gene Ontology）是一个标准化的基因功能分类体系。它把基因的功能分成三个大类：

**BP（Biological Process）：** 基因参与的生物学过程，比如"干旱响应"、"光合作用"、"细胞分裂"。这通常是我们最关注的。

**MF（Molecular Function）：** 基因产物的分子功能，比如"转录因子活性"、"蛋白激酶活性"。

**CC（Cellular Component）：** 基因产物在细胞中的位置，比如"细胞核"、"叶绿体"、"细胞膜"。

每个 GO term 有一个唯一编号（如 GO:0009414 = response to water deprivation）和一段文字描述。

### 运行 GO 富集

```r
# ============================================================
# GO 过表达分析（ORA）
# 分别对上调基因和下调基因做分析
# 这比把上调+下调混在一起做更有生物学意义——
# 因为上调和下调通常涉及不同的功能
# ============================================================

# 上调基因列表
genes_up <- res %>%
    filter(padj < 0.05, log2FoldChange > 1) %>%
    pull(Gene)

# 下调基因列表
genes_down <- res %>%
    filter(padj < 0.05, log2FoldChange < -1) %>%

# 为什么用检测到的基因而不是全基因组？
# 因为 RNA-seq 本身就有检测偏好性——没有被检测到的基因
# 不应该参与统计检验的背景计算
background <- res$Gene

cat("输入基因数：\n")
cat("  上调:", length(genes_up), "\n")
cat("  下调:", length(genes_down), "\n")
cat("  背景:", length(background), "\n")
```

⚠️ **踩坑预警：背景基因选错了**

> ORA 中最常见的错误是**背景基因设置不当**。很多人不设 `universe` 参数，让 clusterProfiler 默认用全基因组作为背景——但这是错的。
>
> 正确的背景应该是**你在 RNA-seq 中实际检测到的所有基因**，不是全基因组。因为 RNA-seq 本身就有偏好性——比如低表达基因更不容易被检测到。如果用全基因组做背景，会高估富集的显著性，产生假阳性。
>
> 更极端的错误是把 DEG 列表本身当作背景——这会让几乎所有 GO term 都"显著"，因为分子和分母是同一个东西。

```r
# ============================================================
# 对下调基因做 GO-BP 富集
# ============================================================

ego_down <- enricher(
    gene          = genes_down,
    universe      = background,
    TERM2GENE     = term2gene,
    TERM2NAME     = term2name,
    pAdjustMethod = "BH",
    pvalueCutoff  = 0.05,
    qvalueCutoff  = 0.05
)

cat("下调基因 GO-BP 富集结果：\n")
cat("  显著 GO terms:", nrow(as.data.frame(ego_down)), "\n\n")
head(as.data.frame(ego_down)[, c("Description", "pvalue")], 5)
```

**上调 = 胁迫防御，下调 = 光合作用和生长。** 这就是干旱响应的经典"权衡"模式。

**怎么读气泡图：**

横轴是 GeneRatio——该 GO term 中的差异基因数 / 差异基因总数。越大说明这个功能类别在你的差异基因中占比越高。

纵轴是 GO term 的名称，按显著性排列。

点的大小是绝对基因数（Count），颜色是 p.adjust 值。理想情况下，你想看到的是：**大且深色的点**（基因多且显著）。

💡 **结果太多怎么办：用 simplify() 去冗余**

> GO 的层级结构意味着很多 term 之间高度重叠。比如"response to water deprivation"是"response to abiotic stimulus"的子类——如果前者富集了，后者必然也富集。这会让结果看起来有很多"显著"的 term，但其实说的是同一件事。
>
> `simplify()` 函数可以合并这些冗余的 term：
>
> ```r
> ego_up_simple <- simplify(ego_up, cutoff = 0.7,
>                            by = "p.adjust")
> ```
>
> `cutoff = 0.7` 表示如果两个 GO term 的基因重叠率超过 70%，只保留更显著的那个。

## Part B：KEGG 通路富集

GO 描述的是通用的功能类别，KEGG 描述的是具体的代谢/信号通路。两者互补。

```r
# ============================================================
# KEGG 通路富集（上调基因）
# 注意：KEGG 使用 NCBI Entrez ID，而不是 TAIR ID
# 需要做 ID 转换
# ============================================================

# 从 annotation_map 中提取 Entrez ID
gene_map <- annotation_map[
    annotation_map$ensembl_gene_id %in% genes_up &
        !is.na(annotation_map$entrezgene_id),
    c("ensembl_gene_id", "entrezgene_id")
]
gene_map <- unique(gene_map)

cat("ID 转换：", nrow(gene_map), "/", length(genes_up),
    "基因成功转换\n")

# KEGG 富集
# organism = "ath": 拟南芥的 KEGG 物种代码
ekegg_up <- enrichKEGG(
    gene         = gene_map$entrezgene_id,
    organism     = "ath",
    pvalueCutoff = 0.05
)

cat("\nKEGG 通路富集结果：\n")
head(as.data.frame(ekegg_up)[, c("Description", "pvalue")], 8)
```

```
📊 输出：
ID 转换： 103 / 103 基因成功转换

KEGG 通路富集结果：
  无显著结果
```

KEGG 结果和 GO 结果讲述的是同一个故事的不同侧面：植物激素信号转导（ABA 通路）、苯丙烷类代谢（细胞壁防御）、谷胱甘肽代谢（抗氧化）——全部是干旱胁迫的经典响应通路。

⚠️ **踩坑预警：KEGG 在线查询可能失败**

> clusterProfiler 的 `enrichKEGG()` 默认需要联网查询 KEGG 数据库。如果你的服务器网络不好或者 KEGG 服务器繁忙，可能会报错或超时。
>
> 解决方案有两个：
>
> 1. 提前下载 KEGG 数据库到本地（用 `download_KEGG()` 函数）
> 2. 用 `enrichPathway()`（来自 ReactomePA 包）替代——Reactome 是另一个通路数据库，不需要联网查询

## Part C：GSEA——不设门槛的富集分析

ORA 做完了。现在我们做 GSEA——它不需要 DEG 列表，用的是全部基因的排名。

```r
# ============================================================
# Step: 准备 GSEA 输入——按统计量排名的全部基因列表
# 用 DESeq2 的 stat 列排序（正值=上调，负值=下调）
# 比用 log2FC 更好，因为 stat 同时考虑了变化幅度和变异度
# ============================================================

# 去掉 NA 值，按 stat（DESeq2 的 Wald 统计量）排序
gene_list <- res %>%
    filter(!is.na(stat)) %>%
    arrange(desc(stat))

# 构建命名向量（GSEA 的输入格式）
gsea_input <- gene_list$stat
names(gsea_input) <- gene_list$Gene

cat("GSEA 输入：\n")
cat("  基因数:", length(gsea_input), "\n")
cat("  范围:", round(min(gsea_input), 2), "到",
    round(max(gsea_input), 2), "\n")
cat("  前 3:", head(gsea_input, 3), "\n")
cat("  后 3:", tail(gsea_input, 3), "\n")
```

```
📊 输出：
GSEA 输入：
  基因数: 19769
  范围: -9.61 到 11.7
  前 3: 11.69884 11.12461 8.859053
  后 3: -7.15102 -8.290804 -9.611816
```

注意这个列表包含了 18,890 个基因——远多于 ORA 的 3,400 个 DEG。那些 padj > 0.05 但仍然有一定变化趋势的基因，在 GSEA 中也能贡献信息。

```r
# ============================================================
# 运行 GSEA（GO-BP）
# ============================================================

gsea_go <- GSEA(
    geneList     = gsea_input,
    TERM2GENE    = term2gene,
    TERM2NAME    = term2name,
    minGSSize    = 15,       # 最小基因集大小（太小的不稳定）
    maxGSSize    = 500,      # 最大基因集大小（太大的不具体）
    pvalueCutoff = 0.05,
    verbose      = FALSE,
    seed         = TRUE,
    by           = "fgsea"
)

cat("GSEA GO-BP 结果：\n")
cat("  显著基因集:", nrow(as.data.frame(gsea_go)), "\n")

# 按 NES（Normalized Enrichment Score）排序
# NES > 0: 在上调基因中富集（干旱处理上调的通路）
# NES < 0: 在下调基因中富集（干旱处理下调的通路）
gsea_df <- as.data.frame(gsea_go) %>%
    arrange(desc(NES))

cat("\n上调方向 top 5：\n")
print(gsea_df[1:5, c("Description", "NES", "p.adjust")])
cat("\n下调方向 top 5：\n")
print(tail(gsea_df, 5)[, c("Description", "NES", "p.adjust")])
```

```
📊 输出：
GSEA GO-BP 结果：
  显著基因集: 0

上调方向 top 5：
  无显著结果

下调方向 top 5：
  无显著结果
```

在这次 500k-spot smoke 子集上，GSEA 没有得到显著基因集；这说明它对排序稳定性和测序深度更敏感。ORA 仍然清楚地区分出了上调的胁迫响应和下调的光合作用，但更完整的 GSEA 结论需要在全量数据上再验证一次。

想象你从排名列表的最左边（最上调的基因）开始往右走。每遇到一个属于"水分匮乏响应"的基因，enrichment score 就往上走一步；每遇到一个不属于的基因，就微微下降。

如果这个基因集的基因均匀分布在排名列表中（即和干旱处理无关），那么曲线会在零附近随机波动。但我们看到曲线从左端开始就持续上升——说明"水分匮乏响应"的基因大量集中在排名靠前的位置（最上调的那些基因中），这正是 GSEA 要检测的富集信号。

## ORA vs GSEA：什么时候用什么

我们已经用两种方法做了富集分析。让我们整理一下它们的适用场景：

| | ORA | GSEA |
|---|---|---|
| 输入 | DEG 列表（需要阈值） | 全部基因的排名列表（不需要阈值） |
| 优势 | 简单直观，容易解释 | 不依赖阈值，能发现温和的集体效应 |
| 劣势 | 依赖 DEG 阈值，丢失连续信息 | 结果不如 ORA 直观，解释需要经验 |
| 适合 | DEG 数量多（>500），效应强 | DEG 数量少，或想看全局趋势 |
| 论文中 | 通常作为主要展示 | 通常作为补充验证 |

💡 **一个务实的写论文策略**

> 在论文中同时展示两者的结果，用 ORA 的气泡图作为主图（直观），用 GSEA 的 enrichment plot 作为补充图（严谨）。如果审稿人质疑你的 DEG 阈值选择，GSEA 可以作为"阈值无关"的证据回应。

## 本篇小结

这一篇我们做了三件事：用 ORA 分别对上调和下调基因做了 GO/KEGG 富集分析，用 GSEA 做了无阈值的全基因组富集分析，并且学会了怎么读气泡图和 enrichment plot。

核心结论：干旱处理导致拟南芥激活了胁迫防御通路（水分响应、ABA 信号、抗氧化），同时抑制了光合作用和生长相关通路。这和几十年来的干旱胁迫研究完全吻合。

更重要的收获是方法层面的：

**ORA 和 GSEA 回答不同的问题。** ORA 问"我的 DEG 里有没有特定功能的基因过多"，GSEA 问"特定功能的基因是不是倾向于在变化最大的那些基因中"。两者互补，不是非此即彼。

**背景基因的选择至关重要。** ORA 的背景应该是你实际检测到的全部基因，不是全基因组，更不是 DEG 列表本身。

**上调和下调基因应该分开做 ORA。** 混在一起做会丢失方向信息——你不知道一个通路到底是被激活了还是被抑制了。GSEA 天然包含方向信息（NES 正负），所以不需要分开做。

## 下一篇预告

通路富集告诉我们"这些基因在功能上属于什么类别"。但基因不是孤立行动的——它们组成网络，共同应对环境变化。

下一篇我们进入 WGCNA（加权基因共表达网络分析）。WGCNA 不关心"哪些基因是显著的"，它关心的是"哪些基因的表达模式总是同步变化"。通过识别这些共表达模块（module），我们可以发现传统差异分析无法揭示的基因协作关系——比如哪一组基因共同被某个转录因子调控，哪一组基因和干旱胁迫表型最相关。
