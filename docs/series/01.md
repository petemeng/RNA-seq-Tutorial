## 先搞清楚一件事：RNA-seq 的目的是什么

在讲任何技术细节之前，我们先用最朴素的语言说清楚 RNA-seq 在做什么。

你已经知道，基因是生命活动的"蓝图"。但一个细胞在某个时刻并不会使用全部基因——它只会"打开"一部分基因来制造蛋白质，而"关闭"另一部分。这个过程的中间产物就是 RNA：基因被"转录"成 RNA，RNA 再被"翻译"成蛋白质。

**RNA-seq 做的事情，就是给一个样本里所有正在被使用的基因拍一张"快照"。**

通过这张快照，我们可以回答一个核心问题：**在不同条件下，哪些基因的活跃程度发生了变化？**

比如：用药前和用药后，肿瘤里哪些基因被激活了？干旱胁迫下，植物根部哪些基因的表达量上升了？感染了病原菌的番茄和健康番茄相比，免疫相关基因有什么变化？

这些问题的答案藏在测序数据里。而 RNA-seq 分析，就是把这些答案一步步挖出来的过程。

## 那分析流程长什么样？

如果把 RNA-seq 分析比作做菜，大致可以分成这几步：

**第一步：买菜（获取数据）。** 你的原始数据是测序仪产出的 FASTQ 文件——可以理解为一个巨大的文本文件，里面记录了几千万条短 DNA 片段（叫做 reads）的序列信息。这些 reads 就是你样本中 RNA 分子的"碎片化抄写"。

**第二步：洗菜（质量控制）。** 不是所有 reads 都是高质量的。有些 reads 的末端序列质量很差（测序仪"犹豫"了），有些 reads 带着实验过程中残留的接头序列（adapter）。质控这一步就是把这些低质量的部分清理掉，确保后续分析用的数据是干净的。

**第三步：切配（比对与定量）。** 现在你手上有几千万条短序列，但你还不知道每条序列来自哪个基因。"比对"（alignment）就是把这些序列放回参考基因组上，找到它们的来源位置。比对完成后，"定量"（quantification）就是数一数：每个基因上落了多少条 reads？数量越多，说明这个基因在这个样本中越活跃。

**第四步：上桌（差异分析）。** 有了每个基因在每个样本中的 reads 计数，我们就可以比较了：处理组和对照组之间，哪些基因的计数有统计学显著的差异？这一步会给你一份差异表达基因（DEG）的列表——这通常就是你写论文时最需要的结果。

## 这个系列和其他教程有什么不同

你可能已经搜过一些 RNA-seq 教程了。也许看了几篇公众号推文，也许翻了翻英文的 Bioconductor 教程。你发现有的教程上来就甩命令，有的教程长篇累牍讲数学——但都没有让你觉得"我跟着做就能学会"。

问题出在哪里？

大多数教程犯的一个共同错误是：**它们假设你已经知道为什么要做这一步，然后只教你怎么做。**

比如你可能在某篇教程里看到这样的内容：

> "使用 STAR 进行比对：
> `STAR --runMode alignReads --genomeDir /path/to/index --readFilesIn sample_R1.fastq.gz sample_R2.fastq.gz`"

然后就没了。你把命令复制过去，可能跑成功了，也可能报错了。但不管哪种情况，你都不知道：这条命令到底做了什么？`--genomeDir` 应该指向什么？如果报错了应该先检查什么？

**我们这个系列的写法完全不同。每一步都先回答"为什么"，再回答"怎么做"，最后回答"怎么确认做对了"。**

具体来说，我们会这样讲同一个步骤：

> 比对（alignment）的本质，是把你的 reads "放回"到参考基因组上。你可以想象一个巨大的拼图：参考基因组是完成的拼图全貌，而每条 read 是一个小拼图块。比对软件的工作就是找到每个小块应该放在拼图的哪个位置。
>
> 我们在这个教程中选择 STAR 作为比对工具。为什么是 STAR 而不是其他工具？因为 STAR 在处理剪接位点（splice junction）方面表现优异——RNA-seq 的 reads 经常跨越外显子-外显子的连接处，STAR 对这种情况的处理比大多数比对工具更准确。这一点已经被多项独立 benchmark 验证过。

看到区别了吗？前者教你敲命令，后者教你理解整件事。

## 一个关于"可复现"的小故事

在正式开始之前，还有一件事需要提前说清楚。

去年我把一套 RNA-seq 分析流程发给了一位新同事。流程文档很完整，每条命令都写了。他照着跑了一遍，结果也出来了。两周后我在另一台服务器上重新跑了一次同样的流程——差异表达基因数量差了将近三倍。

排查了一整天，最后发现问题出在一个你可能想不到的地方：比对软件 STAR 的版本差异。两台服务器上的 STAR 版本只差了一个小版本号（2.7.9a vs 2.7.10b），但在处理 splice junction 时有一个默认参数的变化，积累到最终的结果就是完全不同的 DEG 列表。

为什么讲这个故事？因为它说明了一件事：**能跑出结果，不等于结果是对的。**

在这个系列里，我们不仅教你怎么跑分析，还教你怎么确认结果是可靠的。每一篇都会包含：

- **检查点**：这一步完成后，怎么验证输出是合理的
- **版本记录**：哪些工具的版本会影响结果，怎么记下来
- **排错指南**：如果出了问题，从哪里开始查

这些内容不是"高级技巧"，而是从第一天就应该养成的习惯。我们会在每一篇里自然地融入这些实践，让你从一开始就建立正确的工作方式。

## 那我现在需要准备什么？

如果你决定跟着这个系列学下去，你需要准备的东西其实很少：

**一台能用 Linux 命令行的电脑。** 可以是你实验室的服务器，也可以是一台装了 WSL（Windows Subsystem for Linux）的 Windows 电脑，甚至可以是 Mac 的终端。如果你从来没用过命令行，不用慌——下一篇会从最基本的操作开始讲起。

**一点耐心。** 生物信息分析的学习曲线在最初阶段是陡峭的。你会遇到报错，会看到看不懂的输出，会一度觉得"这也太复杂了"。这是完全正常的。每一篇教程的节奏都是按照初学者设计的，我们会在你可能卡住的地方提前给出解释。

**不需要编程基础。** 我们会用到一些命令行操作和简单的脚本，但每一条命令都会有详细的解释。你不需要"会编程"，你只需要愿意跟着敲一敲键盘。

💡 **关于 Python 和 R**

> 你可能听说 RNA-seq 分析需要用 Python 或 R。没错，差异分析阶段（第 5 篇）我们会用到 R 语言中的 DESeq2 包。但在那之前的所有步骤都是在 Linux 命令行下完成的，不涉及编程。
>
> 到了需要用 R 的时候，我们会从"怎么安装 R"开始讲。所以现在完全不需要提前学 R。

## 后续各篇预告

整个系列按照分析流程的自然顺序排列。每一篇只解决一个阶段的问题：

**第 2 篇：拿到数据——测序数据是什么样的，从哪里下载。** 这一篇会解释 FASTQ 文件的结构（它没有听起来那么复杂），然后手把手带你从公共数据库下载一个真实的测序数据集。我们选了一个小巧但完整的数据集，几分钟就能下完，不会在第一步就卡在"数据太大下不动"的问题上。

**第 3 篇：检查数据——测序质量好不好，需不需要清洗。** 教你生成和阅读质控报告。重点不是"怎么运行 fastp"，而是"报告里那些花花绿绿的图到底在告诉你什么"。

**第 4 篇：分析数据——比对到参考基因组，数一数每个基因有多少 reads。** 这是计算量最大的一步，也是最容易出错的一步。我们会详细讲参考基因组和注释文件的选择、比对的基本原理、以及怎么判断比对结果是否正常。

**第 5 篇：得出结论——差异表达分析，找到不同条件下变化的基因。** 这一篇的目标是让你不仅能跑出一张 DEG 列表，还能判断这张列表是否可信。我们会讲 DESeq2 的基本逻辑（不讲公式，讲直觉），以及怎么做火山图和热图来展示你的发现。

**第 6 篇：整理输出——怎么把分析过程变成别人也能看懂、你自己也能复现的记录。** 这一篇关注的不是新的分析步骤，而是"怎么把前面做的事情整理好"——包括结果文件的组织、分析流程的记录、以及图表的美化。
